# 开发规则

---

## 一、通用原则（所有场景适用）

### 1. 分支管理
- **永远不要直接提交到主干分支**（main、develop）
- 如果当前处于主干分支，先创建新分支再进行修改

### 2. 功能保护原则
- **做任何改动时，必须思考是否会破坏已有功能**
- 如果改动会影响已有功能，**必须先询问开发者**
- 不得擅自破坏或删除已有功能，除非开发者明确要求

### 3. 规划阅读原则
- **每次开始新任务前，先阅读 [docs/overview.md](docs/overview.md) 了解项目入口**
- 根据任务类型查看 overview.md 中的任务路由表，找到对应设计文档
- 避免每次从头扫描源码，保持一致性

### 4. 文档记录原则

- **修改代码后，必须检查相关设计文档是否需要同步更新**（如新增接口、变更行为、调整设计等）
- 代码变更与文档不一致会误导后续开发

#### 设计文档的职责
- **关键设计决策必须写入专门的设计文档**
- **目的**：防止后续修改破坏已有设计

### 5. 代码规范
- **不要修改明确禁止修改的代码**（如 C++ 核心代码，除非明确要求）
- 遵循项目现有的代码风格
- 遵循现代C++风格、C++ Core Guidelines
- 修改前先阅读相关代码，理解现有实现

### 6. 确认机制
- **任何阶段的关键产出，都需要开发者确认后才能继续**
- 等待明确回复（如"好的"、"可以"、"确认"等）
- **禁止**：假设开发者会同意而跳过确认

### 7. 代码格式化
- **每次提交前必须运行格式化**：`./scripts/check-format.sh --fix`
- 确保所有代码变更通过格式检查后再提交
