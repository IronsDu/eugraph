# 开发规则

---

## 一、通用原则（所有场景适用）

### 1. 分支管理
- **永远不要直接提交到主干分支**（main、develop）
- 如果当前处于主干分支，先创建新分支再进行修改

### 2. 功能保护原则
- **做任何改动时，必须思考是否会破坏已有功能**
- 如果改动会影响已有功能，**必须先询问开发者**
- 不得擅自破坏或删除已有功能，除非开发者明确要求

### 3. 规划阅读原则
- **每次开始新任务前，先阅读 plan.md 及其中引用的相关文档**
- 了解项目的整体规划、架构设计和已有模块
- 避免每次从头摸索，保持一致性

### 4. 文档记录原则

#### plan.md 的定位
- **plan.md 专用于计划和进度跟踪，不承载详细设计内容**
- 记录内容：
  - 整体计划安排（做什么）
  - 当前正在进行的任务
  - 已完成的任务（标记完成状态）
  - 引用具体的设计文档（如 architecture.md等）
- **禁止**：在 plan.md 中写详细设计细节

#### 设计文档的职责
- **关键设计决策必须写入专门的设计文档**（如 architecture.md等）
- 记录内容：
  - 新增接口的用途和参数
  - 与现有接口的区别
  - 关键技术点和设计决策
- **目的**：防止后续修改破坏已有设计

### 5. 代码规范
- **不要修改明确禁止修改的代码**（如 C++ 核心代码，除非明确要求）
- 遵循项目现有的代码风格
- 修改前先阅读相关代码，理解现有实现

### 6. 确认机制
- **任何阶段的关键产出，都需要开发者确认后才能继续**
- 等待明确回复（如"好的"、"可以"、"确认"等）
- **禁止**：假设开发者会同意而跳过确认

### 7. 代码格式化
- **每次提交前必须运行格式化**：`./scripts/check-format.sh --fix`
- 确保所有代码变更通过格式检查后再提交

## 二、大型项目开发流程（从零开始）

**适用场景**：启动一个新的大型项目，需要从整体架构开始设计。

### 流程概览

```
步骤 1: 目标与需求定义 → 等待确认
步骤 2: 技术预研（如需要）→ 等待确认
步骤 3: 架构设计 → 等待确认
步骤 4: 数据模型设计 → 等待确认
步骤 5: 系统分层与模块划分 → 等待确认
步骤 6: 接口契约定义 → 等待确认
步骤 7: 里程碑规划 → 等待确认，然后进入模块开发流程
```

**此阶段聚焦设计，不涉及编码。**

### 各阶段说明

#### 步骤 1: 目标与需求定义
- 明确项目要解决什么问题
- 定义核心功能范围和边界
- 产出：需求文档

#### 步骤 2: 技术预研（如需要）
- 对不确定的技术选型进行 POC 验证
- 评估技术方案的可行性、性能、风险
- 产出：技术选型建议或验证报告

#### 步骤 3: 架构设计
- 确定整体技术栈和架构风格
- 设计系统的核心组件及其关系
- 考虑非功能性需求（性能、安全、可扩展性）
- 产出：architecture.md

#### 步骤 4: 数据模型设计
- 设计核心数据结构和关系
- 考虑数据一致性、完整性约束
- 产出：architecture.md 中的数据章节

#### 步骤 5: 系统分层与模块划分
- 确定系统的层次结构（如表示层、业务层、数据层）
- 划分功能模块，明确各模块职责
- 产出：architecture.md 中的模块章节

#### 步骤 6: 接口契约定义
- 定义模块间的接口规范
- 明确接口的输入、输出、行为约定
- 产出：architecture.md 中的接口章节

#### 步骤 7: 里程碑规划
- 确定项目的交付阶段和优先级
- 规划每个里程碑的目标和范围
- 产出：plan.md 中的里程碑计划

## 三、模块/功能开发流程（在现有架构下）

**适用场景**：在已有架构下开发具体模块或功能。

### 流程概览

```
步骤 1: 设计方案对齐 → 等待确认
步骤 2: 测试用例对齐 → 等待确认
步骤 3: 开始编码实现
步骤 4: 运行测试
步骤 5: 通知开发者验证
```

### 详细说明

#### 1. 设计先行
- 在开发任何功能之前，**必须先进行设计**
- 设计方案需要与开发者对齐，确认无误后才能开始开发
- 如果开发者没有提供设计方案，必须自己思考设计，并与开发者对齐
- **关键**：必须等待开发者明确确认后，才能开始编码

#### 2. 测试驱动
- 在设计方案确认后、开始编码前，**必须先对齐测试用例**
- 测试用例需要与开发者对齐：
  - **如何测试**（测试方法、测试工具）
  - **测试什么**（测试场景、边界条件）
  - **成功标准**（什么算通过）
- **禁止**：编码前未讨论测试用例、边开发边想如何测试
- 只有测试用例通过后，才能通知开发者验证

#### 3. 交付验证
- 编码完成且测试通过后，通知开发者进行验证
- 开发者验证通过后，该任务才算完成，并更新计划状态
