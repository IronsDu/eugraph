# 开发规则

---

## 零-一、每次任务启动前（BLOCKING — 逐项确认后才能改代码）

> 以下每一项都必须在动手前完成。任一项未满足，**禁止修改任何文件**。

- [ ] **分支检查**：当前在 main 或 develop 上？→ **立即创建新分支**，然后再做任何修改
- [ ] **入口阅读**：已阅读 [docs/README.md](docs/README.md) 了解项目结构和任务路由？
- [ ] **设计文档**：已根据任务类型，阅读对应的设计文档？

## 零-二、每次提交前（BLOCKING）

> 以下检查全部通过后才能 `git commit`。

- [ ] **格式化**：通过
- [ ] **编译**：通过
- [ ] **测试**：通过
- [ ] **文档同步**：代码变更涉及接口/行为/文件组织变化时，已检查并更新相关设计文档？

## 零-三、编码实施流程（BLOCKING — 逐项确认后才能继续下一步）

> 以下步骤必须严格按顺序执行，每步完成后必须等待开发者明确确认才能进入下一步。
> 🔴 **违反上述流程将导致严重问题！**

- [ ] **第 1 步：设计方案对齐** → 等待开发者明确确认 ✅
- [ ] **第 2 步：测试用例对齐** → 等待开发者确认 ✅
- [ ] **第 3 步：开始编码实现**（前两步确认通过后才能开始）
- [ ] **第 4 步：运行测试**
  - 测试未通过？→ **自行修复问题**，重复第 4 步直到全部通过
  - 测试全部通过后，才能进入第 5 步
- [ ] **第 5 步：通知开发者验证**（测试全部通过后方可通知）

## 一、通用原则（所有场景适用）

### 1. 功能保护原则
- **做任何改动时，必须思考是否会破坏已有功能**
- 如果改动会影响已有功能，**必须先询问开发者**
- 不得擅自破坏或删除已有功能，除非开发者明确要求

### 2. 代码与文档规范

- **禁止使用裸指针和取地址符号（`&`）**：除非完全无法绕开，否则一律使用智能指针或引用替代裸指针，禁止对变量取地址
- 遵循项目现有的代码风格，遵循现代 C++ 风格、遵循C++ Core Guidelines
- 修改前先阅读相关代码，理解现有实现
- **修改代码后，必须检查相关设计文档是否需要同步更新**（如新增接口、变更行为、调整设计等）
- **关键设计决策必须写入专门的设计文档**，防止后续修改破坏已有设计

#### 物理设计与复杂度控制（参考《大规模C++程序设计》）

- **最小化头文件依赖**：优先使用前置声明，仅在实际使用定义时才 `#include`（如只需指针/引用时无需引入完整定义）
- **编译防火墙**：对内部实现细节使用 Pimpl 或纯接口类隔离，减少头文件暴露，降低编译耦合
- **组件单一职责**：每个 `.h/.cpp` 文件对围绕一个明确的职责，不要把不相关的功能混在同一个文件中
- **依赖方向单一**：模块间依赖必须单向，禁止循环依赖；高层依赖低层，低层不感知高层
- **避免过度抽象**：不要为假设的未来需求设计，三个相似场景再抽象，两个就直接写
- **头文件自包含**：每个 `.h` 文件必须能独立编译，所需依赖自行 include，不依赖调用方的 include 顺序
