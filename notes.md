# 开发规则

---

## 一、通用原则（所有场景适用）

### 1. 分支管理
- **永远不要直接提交到主干分支**（main、develop）
- 如果当前处于主干分支，先创建新分支再进行修改

### 2. 功能保护原则
- **做任何改动时，必须思考是否会破坏已有功能**
- 如果改动会影响已有功能，**必须先询问开发者**
- 不得擅自破坏或删除已有功能，除非开发者明确要求

### 3. 规划阅读原则
- **每次开始新任务前，先阅读 [docs/README.md](docs/README.md) 了解项目入口**
- 根据任务类型查看 docs/README.md 中的任务路由表，找到对应设计文档
- 避免每次从头扫描源码，保持一致性

### 4. 代码与文档规范

- **不要修改明确禁止修改的代码**（如 C++ 核心代码，除非明确要求）
- 遵循项目现有的代码风格，遵循现代 C++ 风格、C++ Core Guidelines
- 修改前先阅读相关代码，理解现有实现
- **修改代码后，必须检查相关设计文档是否需要同步更新**（如新增接口、变更行为、调整设计等）
- **关键设计决策必须写入专门的设计文档**，防止后续修改破坏已有设计

#### 物理设计与复杂度控制（参考《大规模C++程序设计》）

- **最小化头文件依赖**：优先使用前置声明，仅在实际使用定义时才 `#include`（如只需指针/引用时无需引入完整定义）
- **编译防火墙**：对内部实现细节使用 Pimpl 或纯接口类隔离，减少头文件暴露，降低编译耦合
- **组件单一职责**：每个 `.h/.cpp` 文件对围绕一个明确的职责，不要把不相关的功能混在同一个文件中
- **依赖方向单一**：模块间依赖必须单向，禁止循环依赖；高层依赖低层，低层不感知高层
- **避免过度抽象**：不要为假设的未来需求设计，三个相似场景再抽象，两个就直接写
- **头文件自包含**：每个 `.h` 文件必须能独立编译，所需依赖自行 include，不依赖调用方的 include 顺序

### 5. 确认机制
- **任何阶段的关键产出，都需要开发者确认后才能继续**
- 等待明确回复（如"好的"、"可以"、"确认"等）
- **禁止**：假设开发者会同意而跳过确认

### 6. 提交前检查清单

每次提交前必须完成以下检查：

- **格式化**：运行 `./scripts/check-format.sh --fix`，确保通过格式检查
- **构建**：确保代码可正常编译
- **测试**：运行相关测试，确保不破坏已有功能
